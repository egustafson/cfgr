# cfgr cli for out of scm config management

The `cfgr` CLI tool provides a way to manage configuration files that are not directly under version control. It allows you to maintain a source directory of configuration files that can be version controlled, and then push or pull changes to the actual deployed location of those files on your system.

The tool supports the following commands:
- `version`: Print the installed version of `cfgr`
- `diff`: Show differences between the source directory and the target deployed location
- `pull`: Pull changes from the target deployed location into the source directory
- `push`: Push changes from the source directory to the target deployed location

The configuration for `cfgr` is specified in a `.cfgr.yml` file, which defines the target directory, as well as any ignore patterns.  A `.cfgr.yml` file is required in the top directory of a source tree.  The `target` field specifies the deployed location of the configuration files. The `ignore` field allows you to specify patterns for files or directories that should be ignored when performing operations.  Child directories in the source tree are supported, and the directory structure will be mirrored in the target location when pushing changes.  The tool uses `pathspec` to handle ignore patterns, which supports gitignore-style syntax.  

`.cfgr.yml` files are supported in child directories. When present, the child `.cfgr.yml` may not specify a `target` field and cfgr should error if it does. The child `.cfgr.yml` is used to specify additional ignore patterns that are specific to that subdirectory. When performing operations, `cfgr` will combine the ignore patterns from the parent and child `.cfgr.yml` files to determine which files to ignore in that subdirectory. This allows for more granular control over which files are included or excluded in different parts of the source tree.  

Files or file patterns that are listed in the `ignore` field of a `.cfgr.yml` file will be excluded from all operations performed by `cfgr`. This means that when you run the `diff`, `push`, or `pull` commands, any files that match the ignore patterns will not be considered for comparison or copying. This allows you to easily exclude certain files or directories from being managed by `cfgr`, which can be useful for temporary files, logs, or other files that should not be version controlled or deployed.  The ignore patterns are applied recursively, so if you specify an ignore pattern in a parent `.cfgr.yml` file, it will apply to all child directories unless overridden by a child `.cfgr.yml` file. This allows you to set global ignore patterns at the top level of your source tree, while still allowing for specific overrides in subdirectories as needed.

The file ignore patters are applied to both the source and target locations. This means that if a file matches an ignore pattern, it will be ignored in both the source directory and the target deployed location. For example, if you have a log file that is generated in the target location and you want to ignore it, you can specify an ignore pattern for that log file in your `.cfgr.yml` file. This will ensure that the log file is not considered when performing operations like `diff`, `push`, or `pull`, regardless of whether it exists in the source directory or the target location. This allows you to maintain a clean and focused set of configuration files while still ignoring any temporary or generated files that should not be managed by `cfgr`.

# diff command

The `diff` command compares the files in the source directory with those in the target deployed location and shows any differences. It uses the `difflib` library to generate a unified diff output, which is similar to the output of the `diff -u` command in Unix. The diff output includes the file paths and line numbers for any differences found between the source and target files. This allows you to easily see what changes would be made if you were to push or pull changes between the source and target locations.

The `diff` command has a `--short` option that includes the `-s` flag.  The short option will display the list of files that differ between the source and target, but will not show the actual line-by-line differences. This can be useful for quickly identifying which files have changes without needing to see the specific details of those changes. When the `--short` option is used, the output will simply list the file paths of any files that differ between the source and target locations.

The `diff` command also has a `--no-ignore` option that includes the `-I` flag.  By default, the `diff` command will ignore any files that match the ignore patterns specified in the `.cfgr.yml` file. However, when the `--no-ignore` option is used, the `diff` command will include all files in the comparison, regardless of whether they match any ignore patterns. This can be useful for debugging or for seeing all differences between the source and target locations without any exclusions.

The --no-ignore option can be used in combination with the --short option to see a list of all files that differ between the source and target locations, including those that would normally be ignored. This allows you to get a complete picture of all differences, even for files that are typically excluded from consideration due to ignore patterns.

# push and pull commands

The `push` command copies files from the source directory to the target deployed location, while the `pull` command copies files from the target deployed location back to the source directory. Both commands will only copy files that differ between the source and target locations, as determined by the `diff` command. This means that if a file in the source directory is identical to the corresponding file in the target location, it will not be copied when using either `push` or `pull`. This helps to minimize unnecessary file operations and ensures that only actual changes are propagated between the source and target locations.

The `push` and `pull` commands can be used with explicit file paths to specify which files to copy. When you provide specific file paths as arguments to the `push` or `pull` commands, only those files will be copied between the source and target locations, regardless of whether they differ or not. This allows you to selectively push or pull specific files without needing to compare the entire directory tree. For example, if you only want to push a single configuration file from the source to the target location, you can run `cfgr push path/to/file` to copy just that file.

Both `push` and `pull` also support a `--force` option which does not have a short flag. The `--force` option requires explicit file paths; it can not be used with the `push` or `pull` commands without specifying which files to operate on. When the `--force` option is used, the specified files will be copied from the source to the target location (for `push`) or from the target to the source location (for `pull`) regardless of whether they differ or not. This allows you to overwrite files in the target location with those from the source, or vice versa, even if they are identical, which can be useful for ensuring that certain files are updated or for resetting files to a known state.

If the force option is used and the files would otherwise be ignored due to matching an ignore pattern, cfgr will offer to make changes to the .cfgr.yml file to un-ignore the specified files so that they are tracked in the future. This allows you to easily update your ignore patterns to include files that you want to manage with `cfgr`, even if they were previously excluded. When you run a `push` or `pull` command with the `--force` option and specify files that match ignore patterns, `cfgr` will prompt you to confirm whether you want to modify the `.cfgr.yml` file to remove the relevant ignore patterns for those files. If you choose to proceed, `cfgr` will automatically update the `.cfgr.yml` file to ensure that those files are no longer ignored in future operations. This provides a convenient way to manage your ignore patterns and ensure that important configuration files are included in your version control and deployment processes.
